# Prompt patterns

## The structure

A production prompt has five pieces — in this order, because it
mirrors how the model processes context:

```
1. Role / persona
2. Task definition
3. Context / inputs
4. Output format / constraints
5. Examples (few-shot)
```

### Example (Anthropic XML)

```
You are a senior security reviewer. Analyze the provided code
diff for exploitable vulnerabilities only.

<task>
Find security issues in the code. Skip style, performance, and
maintainability concerns.
</task>

<context>
<file path="src/api/users.py">
<diff>
...
</diff>
</file>
</context>

<format>
Return a JSON object matching this schema:
{
  "findings": [
    {"severity": "critical|high|medium|low|info",
     "file": "string", "line": "number",
     "issue": "string", "exploit": "string", "fix": "string"}
  ]
}
</format>

<example>
Input diff: <minimal example with a SQL injection>
Output: {"findings":[{"severity":"critical","file":"api.py","line":42,"issue":"SQL injection via unparameterized query","exploit":"email=a' OR 1=1--","fix":"Use parameterized query"}]}
</example>
```

### Example (OpenAI messages + tools)

```python
response = client.chat.completions.create(
    model="gpt-4o",
    temperature=0,
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": format_diff(diff)},
    ],
    tools=[{
        "type": "function",
        "function": {
            "name": "report_findings",
            "parameters": FINDINGS_SCHEMA,
        },
    }],
    tool_choice={"type": "function", "function": {"name": "report_findings"}},
)
findings = json.loads(response.choices[0].message.tool_calls[0].function.arguments)
```

Tool calls give structured output without needing the model to
produce valid JSON by sheer willpower.

## Zero-shot vs. few-shot

- **Zero-shot** — just instructions. Start here.
- **Few-shot** — 2–5 input/output examples. Add when:
  - The task format is non-obvious.
  - You want a specific style / terminology.
  - Zero-shot eval scores below threshold.

### Few-shot done well

- **Diverse** examples (edge cases, not just easy ones).
- **Ordered** so the most representative is last (recency bias
  exists).
- **Consistent** format matching the desired output exactly.
- **Labeled** clearly with delimiters (`Input:`, `Output:` or
  XML).

### Few-shot done badly

- Five nearly identical examples.
- Examples showing only happy-path; none for "unknown" / refusal.
- Examples longer than the actual input (token waste).

## Chain-of-Thought (CoT)

Let the model reason before answering. Effective on:

- Multi-step reasoning (math, logic, planning).
- Ambiguous classification where evidence matters.
- Anything the model gets wrong when asked tersely.

### Approaches

1. **"Think step by step"** — simple trigger; still works with
   modern models, but less necessary.
2. **Structured reasoning block** — explicit tag:
   ```
   <thinking>
   ...
   </thinking>
   <answer>
   ...
   </answer>
   ```
3. **Tool use** (code interpreter, calculator) — for tasks where
   computation is required.

### When NOT to use CoT

- Latency-sensitive path. CoT doubles or triples token output.
- Simple classification. Adds no value.
- When you need deterministic short output for a downstream system.

### Self-consistency

Run CoT N times at temperature > 0, take majority vote. Improves
correctness on reasoning-heavy tasks. Expensive; use in offline
pipelines, not user-facing.

## Tool use / function calling

Let the model call tools: DB query, web search, calculator, code
execution.

### Definition (OpenAI style)

```json
{
  "name": "lookup_user",
  "description": "Look up a user by email address",
  "parameters": {
    "type": "object",
    "properties": {
      "email": {"type": "string", "format": "email"}
    },
    "required": ["email"]
  }
}
```

### Pattern

1. Model sees available tools.
2. On relevant user query, model emits a `tool_call` with args.
3. Your code executes, returns result.
4. Model continues with result included in context.
5. Final answer to user.

### Rules

- **Describe tools precisely.** The tool description is a prompt.
- **JSON schema for parameters.** Reject malformed calls; retry.
- **Small set of tools.** >20 tools → model confuses them; route
  first.
- **Idempotent tools** where possible.
- **Never execute tool code without validation.** Shell / SQL
  generated by the model is an injection risk — pattern-match /
  allowlist.

## ReAct pattern

```
Thought: I need to find the user's most recent order.
Action: lookup_order(email="x@y.com")
Observation: Order #1234, 2024-09-01
Thought: I have enough info.
Answer: Your latest order is #1234 from Sep 1.
```

Frameworks: LangChain agents, OpenAI Swarm, LlamaIndex agents,
Anthropic tool use orchestration. For simple cases, native
function calling is cleaner than ReAct.

## Structured output

### OpenAI

- **JSON mode** (`response_format={"type":"json_object"}`): guarantees
  valid JSON; schema still "optional".
- **Structured Outputs** (`response_format={"type":"json_schema", ...}`):
  guarantees conformance to your schema.
- **Tool calling with forced tool**: same effect.

### Anthropic

- Ask in the prompt; parse with retry.
- Prefilling the assistant turn with `{` kicks JSON output.
- Claude 3.5 Sonnet 2024-10-22+ supports tool use well.

### Gemini

- `response_schema` parameter accepts a Pydantic-style schema.
- Automatic validation.

### Universal pattern

```python
for attempt in range(3):
    raw = call_model(prompt)
    try:
        parsed = schema.parse(raw)
        return parsed
    except ValidationError as e:
        prompt = prompt + f"\n\nLast attempt failed: {e}. Retry."
raise RuntimeError("model couldn't produce valid output after 3 tries")
```

## Delimiters

Always wrap untrusted / multi-part content:

```
<document>
...user-supplied content...
</document>

<instructions>
...your instructions...
</instructions>
```

Benefits:

- Model distinguishes trusted from untrusted.
- Debugging easier: you can see exactly what was passed.
- Injection defenses (see `injection-and-evals.md`).

## Concision

Modern models don't need flowery prompts. Trim:

- "Please kindly" → cut.
- "Think carefully" → cut unless measured helpful.
- "You are a highly intelligent AI" → cut.
- Long preamble → cut to the task.

Token reduction = cost + latency reduction. Prove your prompt
needs each sentence via eval.

## Negative instructions

LLMs handle "do X" better than "don't do Y". Rephrase:

- "Don't use markdown" → "Respond in plain text."
- "Never respond with JSON" → "Respond in a single English
  sentence."
- "Do not mention competitors" → "Focus exclusively on our
  product."

When negatives are necessary, also state the positive.

## Model-specific tricks

### Claude (Anthropic)

- **XML tags** — `<instructions>`, `<examples>`, `<context>`.
- **`<thinking>`** for CoT — visible reasoning.
- **System via `system:` parameter** — not embedded in messages.
- **Prefill assistant turn** — e.g., start with `{` to enforce JSON.
- **Long context** — up to 200k tokens on Claude 3; use for docs.

### GPT (OpenAI)

- **System / user / assistant roles** consistently.
- **Tools** for structured output.
- **Temperature 0** for deterministic; don't also lower top_p.
- **Seed** for partial reproducibility (not guaranteed).
- **Structured Outputs** for guaranteed schema.

### Gemini (Google)

- **System instructions** + **structured response**.
- Strong context caching; set up for long stable prefixes.
- Function calling.

### Llama / Mistral / open models

- **Chat template matters** — use the same tokenizer / template
  the model was trained with.
- Instruct variants are finer-tuned for following; use them.
- Stop sequences per template.

## Prompt caching

Anthropic and OpenAI support caching the stable system prefix.
Benefits:

- 10× cost reduction on cached prefix.
- Lower latency.

### Anthropic

```python
{
    "role": "system",
    "content": [
        {"type": "text", "text": STATIC_PREFIX, "cache_control": {"type": "ephemeral"}},
    ],
}
```

Cache hit if the prefix matches; cache TTL ~5 min.

### OpenAI

Automatic on system prompts > 1024 tokens; discount on repeat.

### Pattern

- Stable instructions + examples first (cached).
- User input last (uncached).
- Gives 10× speedup on chatty use cases (repeated queries to the
  same assistant).
